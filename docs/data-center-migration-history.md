# Data Center Migration History

Applied migrations are immutable records. Do not edit a migration after it has been applied to a shared database. Add a new migration for every correction, then update `supabase/migrations/MANIFEST.json` in the same commit.

On 2026-07-24, the after-sales history was restored from the exact `statements` arrays stored in Supabase project `osmjevtynywbkokzejcp`. Supabase stores each of these migrations as one statement. The local files use UTF-8 LF line endings, while remote comparison first converts CRLF to LF and collapses all final blank lines to one LF. This normalization is implemented and unit-tested in `scripts/verify-migration-manifest.mjs`; SHA-256 still protects every byte of the local file.

Migration 018 rejects a null requested refund amount; 019 is the distinct nullability correction; 020 adds the cumulative refund allocation lock. Migration 021 is the first corrective migration after that restored history. Migration 022 adds the restricted remote-checkpoint RPC used by the Vercel deploy gate. `npm run db:migrations:verify` validates every local SHA-256 plus the three repository-recorded normalized MD5 checkpoints. It is an offline repository integrity check only: it cannot prove that the live Supabase history still matches.

Run `npm run db:migrations:verify` to catch a changed, missing, or unlisted local migration. It is an offline repository check, not evidence of the live database.

`vercel.json` runs the single `scripts/vercel-build.mjs` command. It gives the selected remote-verification credential pair only to the preflight process, then starts `next build` with service-role, management, and database credentials removed. The preflight fails closed: it runs the remote migration truth check first, then the local manifest, lexical bootstrap, disposable PGlite, unit-test, and typecheck checks. It intentionally does not invoke `next build`; the wrapper performs that one build only after preflight passes. Do not replace the Vercel build command with the package default, or the release gate and credential isolation can be bypassed.

The preferred Vercel credential route reuses the server-only project configuration: `NEXT_PUBLIC_SUPABASE_URL` and the secret `SUPABASE_SERVICE_ROLE_KEY`. Migration 022 provides `public.get_applied_migration_checkpoints()`, a fixed SECURITY DEFINER RPC that returns only version, name, statement count, and normalized MD5 for migrations 018-020. It has a fixed `public, supabase_migrations, pg_temp` search path, accepts no arguments, returns no SQL text, and is executable only by `service_role` and `postgres`. The service-role key is supplied only to the remote-verifier child process, never to local smoke, API audit, Next build, or Next start children.

The fallback Vercel route requires `SUPABASE_PROJECT_REF` and a masked fine-grained `SUPABASE_ACCESS_TOKEN` with Supabase Management API `database_read` permission. It uses the read-only Management API to compare the version, name, statement count, normalized statement MD5, and canonical statement text. The management token is supplied only to that verifier child. If neither complete credential pair is configured, a Vercel build fails before compiling the application. Never print either secret in build logs.

`npm run production:verify` is a separate post-deploy live-site audit for `boxsofa.eu`, `www.boxsofa.eu`, SEO, API authorization, and readiness; it also performs the remote migration check first. `npm run production:verify:local` is an offline local build, smoke, and API audit. It uses a dynamically reserved loopback port and a per-process readiness nonce; the health endpoint emits that nonce only when `BOXSOFA_LOCAL_VERIFY=1` and never in a Vercel deployment. It must never be treated as a deployment substitute.

Run `npm run db:bootstrap:validate` for the fast lexical precheck, then `npm run db:bootstrap:execute` for a disposable, in-memory PGlite PostgreSQL execution of the full `supabase/schema.sql` plus catalog assertions. The execution command loads real `pgcrypto`; it stubs only the Supabase `auth.users`, `auth.uid()`, `auth.role()`, `anon`/`authenticated`/`service_role` roles, and empty `supabase_realtime` publication that the bootstrap references. It does not validate Supabase Auth or Realtime service behavior, and it never contacts production or creates a cloud branch.
